# Generate Dockerfile

Once you have identified your tool’s functionalities and dependencies, and created sample tool action scripts, the next step is to containerize your tool using Docker. Containerization provides portability, consistency, and scalability, making it easier to run your tool reliably across different nodepools within the Microsoft Discovery platform.

## Prerequisites

Before creating Docker container images, ensure your local development environment has the following components installed and configured:

### Required Software

1. **Docker Desktop** (Recommended) or Docker Engine
   - Version 20.10.0 or later
   - Download from [Docker Official Website](https://www.docker.com/products/docker-desktop/)
   - Ensure Docker daemon is running

2. **Docker CLI Tools**
   - Verify installation: `docker --version`
   - Verify Docker daemon: `docker info`

3. **Text Editor or IDE**
   - VS Code (recommended with Docker extension)
   - Any editor capable of editing Dockerfile syntax

4. **Azure CLI** (if pushing to Azure Container Registry)
   - Version 2.0.80 or later
   - Login configured: `az login`

## Directory Structure

Before creating the Dockerfile, organize your project structure as follows:

```text
6-solutions/tools-and-models/molToolkit/
    ├── Dockerfile                      # Container definition
    ├── README.md                       # Documentation
    ├── MolToolkit-tool-definition.yaml # Tool configuration
    ├── MolToolkit-agent-definition.yaml # Agent configuration
    └── app/                            # Core implementation scripts
        ├── molecular_utils.py          # Core molecular analysis functions
        ├── io_utils.py                 # Utilities for I/O operations
        ├── get_low_energy_conformer.py # Conformer generation logic
        ├── mol_functional_groups.py    # Functional group analysis
        └── mol_hazardous_groups.py     # Hazardous group prediction
```

The directory structure represents a template for containerized tool development. The core components are the script files, Dockerfile, and requirements.txt file, which define the tool's functionality and dependencies.

In this example:

- The **input/** folder contains sample input files for testing purposes
- The **output/** folder is where results and log files are generated
- The **custom_scripts/** folder includes example scripts that demonstrate custom functionality

When deploying this tool on the Microsoft Discovery platform, these folders represent mount points to the Docker container:

- Input and output directories are mounted to provide data and collect results
- Custom scripts can be dynamically generated by an Agent and mounted to the `/scripts` path within the container

This approach allows for flexible tool execution while maintaining a consistent interface for the Discovery platform.

## Step-by-Step Docker Image Creation

While the `molToolkit` sample already includes a pre-configured Dockerfile in its directory, this section explains how to create one from scratch for your own tools. This provides insight into the containerization process and best practices for Microsoft Discovery platform tools.

### Step 1: Navigate to Tool Directory

Start by changing to your tool's working directory:

```bash
cd <tool-name>
```

This directory should contain all the components identified in the dependency analysis phase, similar to the structure shown in the Directory Structure section above.

### Step 2: Create the Dockerfile

Create a new `Dockerfile` (no file extension) in the `<tool-name>` directory if one doesn't already exist. The structure will vary based on your tool type and dependencies. Below is a template you can customize:

#### Sample Dockerfile for molToolkit (Python based Tool)

```dockerfile
# Dockerfile for MolToolkit
# Base image includes conda-forge channel and mamba tooling
FROM condaforge/mambaforge:latest

WORKDIR /app

# Avoid interactive prompts during apt installs
ENV DEBIAN_FRONTEND=noninteractive

# (Optional) makes RUN lines easier to read + fails on pipe errors
SHELL ["/bin/bash", "-o", "pipefail", "-c"]

# ---- System deps (single layer, no upgrade) ----
# Combine update+install+cleanup in one layer to avoid persisting apt lists
RUN apt-get update \
 && apt-get install -y --no-install-recommends \
      ca-certificates \
      gpgv \
      libsm6 \
      libxrender1 \
      tar \
      wget \
 && rm -rf /var/lib/apt/lists/*

# ---- Conda env (single solve using mamba) ----
# Install everything in one transaction to reduce dependency re-solves
RUN mamba create -y -n molenv -c conda-forge \
      python=3.11 \
      ase \
      biopython \
      mdanalysis \
      matplotlib \
      numpy \
      openbabel \
      pyarrow \
      pandas \
      pymatgen \
      pymol-open-source \
      rdkit \
      ruamel.yaml \
      scanpy \
      scikit-bio \
      scikit-learn \
      scipy \
      seaborn \
      sympy \
 && mamba clean -a -y

# ---- pip (ONLY inside molenv; no duplicate global pip install) ----
RUN mamba run -n molenv python -m pip install --no-cache-dir \
      brotli==1.2.0 \
      fonttools==4.60.2 \
      h2==4.3.0 \
      requests==2.32.4 \
      setuptools==78.1.1 \
      urllib3==2.6.0 \
      werkzeug==3.1.4

# Copy code late to maximize Docker layer caching
COPY app/*.py /app/

# ---- Runtime env vars ----
ENV CONDA_DEFAULT_ENV=molenv
ENV CONDA_PREFIX=/opt/conda/envs/molenv
ENV PATH=/opt/conda/envs/molenv/bin:/opt/conda/bin:$PATH
ENV PYTHONPATH=/app
ENV PYTHONUNBUFFERED=1

# Create common mount points / working dirs (single layer)
RUN mkdir -p /input /output /workdir

# Keep container running
CMD ["tail", "-f", "/dev/null"]
```

#### Explanation of the Sample Dockerfile

The sample Dockerfile provided above is designed to containerize a Python-based tool that uses scientific computing libraries including RDKit, ASE, and more. Here's a comprehensive breakdown of its key sections:

- **Base Image**:
    Uses `condaforge/mambaforge:latest` as the base, which is a lightweight and fast Conda distribution with mamba for efficient package management.

- **Working Directory**:
    Sets `/app` as the working directory inside the container, providing a consistent path for all operations and files.

- **Environment Variables**:
    Sets essential environment variables to ensure consistent behavior:

  - `DEBIAN_FRONTEND=noninteractive`: Prevents interactive prompts during package installation

- **Shell Configuration**:
    Uses the `SHELL` instruction with `pipefail` to ensure pipeline errors are caught properly.

- **System Dependencies**:
    Installs essential system packages required for molecular visualization and general operations. The command follows best practices by:

  - Combining update and install commands in a single RUN instruction
  - Using `--no-install-recommends` to reduce image size
  - Cleaning apt lists to minimize container size

- **Conda Environment Setup**:
    Creates a dedicated environment (`molenv`) with comprehensive scientific packages:

  - Python 3.11: Modern, stable version
  - RDKit, ASE, Biopython: Core cheminformatics and molecular modeling libraries
  - Pandas, NumPy, SciPy: Data processing and numerical computing
  - Matplotlib, Seaborn: Visualization libraries
  - PyMOL, OpenBabel: Molecular visualization and format conversion
  - Uses mamba for faster dependency resolution

- **Python Dependencies**:
    Installs additional pip packages within the conda environment:

  - Uses `--no-cache-dir` flag to reduce container size
  - Keeps core dependencies separate from conda-managed packages

- **Application Code**:
    Copies the tool's Python scripts into the container at the `/app/` directory, which serves as the main location for the tool's functionality.

- **Runtime Environment**:
    Configures environment variables for proper conda activation:

  - Sets `CONDA_DEFAULT_ENV` and `CONDA_PREFIX` for the molenv environment
  - Updates `PATH` to include conda binaries
  - Sets `PYTHONPATH` and `PYTHONUNBUFFERED` for proper Python execution

- **Mount Points**:
    Creates standard directories for input/output data mounting (`/input`, `/output`, `/workdir`).

When adapting this template for your own tools, ensure you modify the base image, dependencies, and configuration to match your specific requirements while maintaining these best practices.

> **Note:** This Dockerfile is provided as a sample and is **not production-grade**. It is intended for demonstration purposes only. Before using any container image in production, you should thoroughly review and harden the Dockerfile, and ensure the resulting image is free of security vulnerabilities.
